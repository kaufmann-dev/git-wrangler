# Command Reference

This is an internal implementation reference for maintainers and AI agents. It is not end-user documentation and does not replace generated Cobra help. Keep it focused on behavior that command help does not fully explain: mutation scope, confirmations, dependencies, repository targeting, auth/config usage, concurrency, output shape, and known consistency gaps.

## Behavior Matrix

| Behavior                                         | Commands                                                                                                                                                              |
| ------------------------------------------------ | --------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| Read-only repository scans with up to 32 workers | `status`, `review`, `info`, scan phases of `fix-gitignore`, `remove-secrets`, `rewrite-commits`, `rewrite-dates`                                                      |
| Independent Git mutations with up to 4 workers   | `fetch`, `pull`, normal `push`, commit preparation/creation, `rename-branch`, apply phases of `remove-secrets`, `rewrite-authors`, `rewrite-commits`, `rewrite-dates` |
| Sequential per-repo flow                         | `clone`, `rename-repo`, `license`, `reset`, `fix-gitignore` apply, `untrack` apply, `push --force-unsafe`                                                             |
| History rewrite and origin restore               | `remove-secrets`, `rewrite-authors`, `rewrite-commits`, `rewrite-dates`                                                                                               |
| Returns nonzero on any per-repo failure          | Most multi-repo commands; no-op skips remain successful except declined destructive history rewrites                                                                  |
| Progress reported to stderr                      | Bulk scan/mutation phases that use `newProgress`; ordered summaries and per-repo result blocks remain on stdout/stderr after collection                               |
| Uses Git Wrangler GitHub credentials             | `clone`, `rename-repo`, `doctor`, `config`, `init`                                                                                                                    |
| Uses Git Wrangler AI settings                    | `commit`, `rewrite-commits`, `doctor`, `config`, `init`                                                                                                               |

## Per-Command Notes

### `clone`

`clone` lists repositories through `gh repo list` and clones each repository with `gh repo clone`. It performs an initial one-item listing to distinguish empty result sets from listing failures before creating the destination directory. Existing destination directories are skipped successfully.

### `commit`

`commit` prepares staged context in a temporary index so scanning does not stage changes before the user approves the data-send prompt. If context collection fails for any repository, it stops before API calls and before creating commits.

### `fetch`

`fetch` runs `git fetch origin` for every discovered repository. `fetch --prune` runs `git fetch --prune origin`. Missing or invalid `origin` remotes are per-repository failures and count in the summary.

### `rename-repo`

`rename-repo` is intentionally interactive and sequential. It skips repositories that `gh repo view` cannot identify as GitHub repositories. There is no `--yes`; entered names are passed to `gh repo rename` with `--yes`.

### `reset`

`reset` fetches `origin <current-branch>`, skips detached HEAD and missing remote counterparts, reports ahead/behind counts, warns when the working tree is dirty, and then runs `git reset --hard origin/<branch>` after confirmation.

### `remove-secrets`

`remove-secrets` scans history for a fixed set of sensitive filename/path patterns, prints matched files, and only rewrites repositories with matches. It always passes `--partial --force` to `git-filter-repo`.

### `rewrite-commits`

`rewrite-commits` validates AI settings before scanning repositories. Generation may prompt before sending data; applying generated messages is a separate all-repository confirmation. Old commit messages are not sent as model context.

### `rewrite-dates`

`rewrite-dates` requires at least two commits per target repository. It prints the old-to-new timestamp summary before confirmation, preserves the dominant timezone offset when possible, and distributes rewritten timestamps between the selected or inferred bounds.

### `config`

`config set` accepts plaintext values for non-secret keys only. Secret keys (`github.auth`, `ai.api-key`) must be entered through the prompt and are stored through the credential store. `config unset` only removes stored secret values.

## Consistency Notes

- Required value handling is split: commands that use `requiredStringFlag` prompt interactively and fail when noninteractive or when `--yes` is present, while `config set` fails immediately when a non-secret value is missing.
- Repository targeting is inconsistent by design today: only `info`, `license`, and `rewrite-authors` expose `--repo`; most multi-repo commands always discover under `.`.
- Confirmation semantics differ. Some mutation commands prompt per repository, `commit` prompts only before sending data, and `rewrite-commits` has a generation prompt plus one apply prompt for all listed repositories.
- `--force` does not mean the same thing across commands. It maps to Git pull force, push force-with-lease, or git-filter-repo force depending on the command.
- History rewrite commands generally treat a declined confirmation as a failure, while local maintenance commands such as `fix-gitignore`, `untrack`, `reset`, and `push --force-unsafe` treat declined repositories as skips.
