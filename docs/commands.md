# Command Reference

This is an internal implementation reference for maintainers and AI agents. It does not replace generated Cobra help. Keep it focused on behavior that command help does not fully explain: targeting, mutation scope, confirmations, dependencies, concurrency, JSON, and output shape.

## Repository Targeting

Most repository commands discover Git worktrees below the current directory when `--repo` is omitted. `--repo PATH` means exact single-repository targeting: operate only on `PATH`, do not recurse below it, and fail if `PATH` is not a normal or linked-worktree Git repository. Bare repository directories are rejected.

Users who previously used `info --repo DIR`, `license --repo DIR`, or `rewrite-authors --repo DIR` to discover nested repositories should omit `--repo` and run from the common parent directory instead.

Commands with `--repo`: `activity`, `log`, `pull`, `fetch`, `push`, `commit`, `fix-gitignore`, `license`, `rename-branch`, `reset`, `review`, `untrack`, `remove-secrets`, `rewrite-authors`, `rewrite-commits`, `rewrite-dates`, `rewrite-hours`, `rollback-rewrites`, `info`, `status`, and `rename-repo`.

Commands without `--repo`: `clone`, `init`, `config`, `doctor`, `version`, `completion`, and `help`.

Remote-aware reporting and history rewrite planning is `origin`-centric. `status`, `info`, `review`, `remove-secrets`, `rewrite-authors`, `rewrite-commits`, `rewrite-dates`, and `rewrite-hours` run `git fetch --prune origin` for target repositories before inspecting remote-tracking refs unless `--no-fetch` is set. `rollback-rewrites` is local-only and never fetches.

## Guided Setup And Prompting

Prompting is available only when both stdin and stderr are TTYs. Missing required
values prompt automatically when prompting is available and fail otherwise,
including when `--yes` is present. A command that reaches a final confirmation
without prompting availability fails nonzero and tells the user to pass
`--yes`. `--yes` and `-y` skip confirmations only.

At any active prompt, `Ctrl+C` or interactive EOF/`Ctrl+D` immediately cancels
the command without requiring Enter. Cancellation stops later prompts and work,
prints `SKIP stopped: operation cancelled` exactly once, and exits nonzero. It
is not treated as a declined confirmation or an empty, invalid, or missing
prompt value.

Command-local `--guided` is available on `activity`, `log`, `clone`, `pull`, `fetch`,
`push`, `commit`, `fix-gitignore`, `license`, `rename-branch`, `reset`, `review`,
`untrack`, `remove-secrets`, `rewrite-authors`, `rewrite-commits`,
`rewrite-dates`, `rewrite-hours`, `rollback-rewrites`, `info`, and `status`. It prompts for all command-specific
behavior options, applies answers through Cobra flag setters, and prints the
selected configuration to stderr before execution. It excludes meta flags such
as `--help`, `--version`, `--guided`, `--yes`, and `--json`.

`--guided` fails outside a TTY and cannot be combined with `--json`.
`rewrite-dates --guided` selects the target-range mode before asking for the
applicable range values. `init` and `rename-repo` remain dedicated
interactive workflows and require a TTY. Secret `config set` values also require
a TTY.

## JSON Shape

JSON mode is command-local and intentionally limited. It uses this general shape:

```json
{
  "ok": true,
  "summary": {},
  "repositories": []
}
```

Fatal command errors set `ok` to `false` and include `error.message`. Per-repository failures are represented inside `repositories[]` and make the command exit 1. `config show --json` reports credential sources and booleans only; it must not expose stored secrets.

For `status`, `info`, and `review`, JSON mode silently performs the default `git fetch --prune origin` refresh before inspection. Fetch failures are represented as per-repository `error.message` entries, stderr remains empty, and the command exits 1 if any repository fails. `--no-fetch` skips this refresh and reports from local remote-tracking refs.

## Per-Command Notes

Human output follows `docs/cli-design.md`: progress and prompts on stderr, durable results on stdout after progress closes, actionable per-repo skips/failures, and final `Summary:` lines for bulk commands. JSON output remains unchanged for commands that support `--json`.

### `clone`

`clone` lists repositories through `gh repo list` and clones each repository with `gh repo clone`. It performs an initial one-item listing to distinguish empty result sets from listing failures before creating the destination directory. An empty repository list is a successful no-op with a zero summary. Private and all visibility use that request as their early authenticated validation. Public visibility never resolves or sends Git Wrangler credentials. Transient GitHub transport failures while listing or cloning are retried with short backoff; authentication, permission, not-found, and target-exists failures are not retried. Existing destination directories are skipped successfully. If a retryable clone failure creates a partial target directory that did not exist before the command, Git Wrangler removes that partial target before retrying. When secure credential storage is unavailable, authenticated cloning hides backend errors and directs users to `GIT_WRANGLER_GITHUB_TOKEN`; `--visibility public` continues without authentication. Human output reports GitHub auth source when used, actionable skips/failures, optional small-run success lines, and a cloned/skipped/failed summary.

### `commit`

`commit` validates the configured AI endpoint, model, and credentials before checking Git or discovering repositories. It then prepares staged context in a temporary index so scanning does not stage changes before the user approves the data-send prompt. If context collection fails for any repository, it stops before generation calls and before creating commits. Declining the data-send prompt is a successful no-op. Human output includes a data-send notice before generation calls, API progress, commit-creation progress, actionable skips/failures, and a committed/skipped/failed summary.

### `fetch`

`pull`, `fetch`, `push`, and automatic `git fetch --prune origin` refreshes use a 30-second timeout per repository. Timeouts are per-repository failures and include the repository plus operation, such as `golden-blog: git pull timed out after 30s`. Timed-out subprocesses terminate their child process tree so helper processes cannot hold the command open after the timeout. Local Git operations and history rewrite subprocesses keep the default longer subprocess timeout.

`fetch` runs `git fetch origin` for every target repository. `fetch --prune` runs `git fetch --prune origin`. Transient transport failures are retried with short backoff; missing or invalid `origin` remotes are not retried and count as per-repository failures. Routine success lines are suppressed.

The built-in auto-refresh used by reporting and history rewrite commands is separate from the explicit `fetch` command. It always uses `git fetch --prune origin`, retries transient transport failures, and has no remote selector or public timeout flag.

### `status`

`status` refreshes `origin` before checking clean/dirty and ahead/behind state. With `--no-fetch`, it uses existing local remote-tracking refs. Fetch failures are per-repository failures: human output continues to inspect other repositories, and JSON output includes failure rows without stderr output.

### `activity`

`activity` renders one aggregated GitHub-style calendar without fetching remotes. By default it scans commits reachable from the repository default branch resolved through `origin/HEAD`, preferring the corresponding local branch, plus local or remote-tracking `gh-pages`. When `origin/HEAD` is unavailable, it scans current `HEAD` and prints a warning after progress finishes.

`--all` scans all normal refs except `refs/git-wrangler/*`. Repeatable `--user` filters author names or emails by exact case-insensitive match. `--year` filters by UTC author date without passing Git committer-date range filters. Commit hashes are deduplicated globally across repositories. Failed scans are discarded, printed as stderr error blocks, and make the command exit 1 after rendering the partial aggregate.

Years render newest first with Sunday-first weeks. The default four-level scale is calculated independently per year; `--global-scale` uses the maximum daily count across all rendered years. TTY output uses green activity cells, while plain output uses `.`, `1`, `2`, `3`, and `4`. No matching commits is a successful no-activity result, while an explicitly requested empty year still renders its calendar.

### `log`

`log` scans history without fetching remotes. By default it uses the repository default branch resolved through `origin/HEAD`, preferring the corresponding local branch and falling back to current `HEAD` with a warning when `origin/HEAD` is unavailable. It runs `git log -z` and parses hash, author date, and subject.

Entries from all successful repositories are filtered by author date, Conventional Commit type or `other`, and exact parsed scope, then merged by author date descending. `--limit` applies after that global merge; `--limit 0` is unlimited. Conventional Commit subjects are parsed with the same validation used for AI-generated commit messages. Non-conventional subjects remain visible as `other`.

Human output is one dense table. Multi-repository output includes `Date`, `Repository`, `Hash`, `Type`, `Scope`, and `Subject`; exact single-repository output omits `Repository`. `--summary` prints compact type, scope, breaking, shown-commit, repository, and failure counts before the table. Per-repository Git failures are printed as stderr error blocks, successful repositories still render, and any failure makes the command exit 1.

### `info`

`info` refreshes `origin` before collecting status, branch, ahead/behind, branch list, remotes, commit counts, authors, and largest files. With `--no-fetch`, it uses existing local remote-tracking refs. Fetch failures are per-repository failures in both human and JSON modes.

### `review`

`review` refreshes `origin` before calculating unpushed commits with `HEAD --not --remotes`. With `--no-fetch`, it compares against existing local remote-tracking refs. Fetch failures are per-repository failures in both human and JSON modes.

### `rename-repo`

`rename-repo` is intentionally interactive and sequential. It validates GitHub credentials before discovering repositories. When secure credential storage is unavailable, it hides backend errors and directs users to `GIT_WRANGLER_GITHUB_TOKEN`. It skips repositories that `gh repo view` cannot identify as GitHub repositories. There is no `--yes`; entered names are passed to `gh repo rename` with `--yes`.

### `license`

`license` creates missing `LICENSE` files without confirmation. `license --overwrite` prompts once before replacing all existing files in the candidate set; `license --overwrite --yes` skips only that overwrite confirmation. Human output prints conflicts/skips/failures and a created/overwritten/skipped/failed summary.

### `fix-gitignore`

`fix-gitignore` scans for common generated-file patterns that exist in a repository but are not ignored. It previews repositories with proposed additions, prompts once for the candidate set unless `--yes` is set, then appends missing entries to `.gitignore`. It does not stage files and does not create commits.

### `rename-branch`

`rename-branch` scans all targets before mutation, prints skips for missing source branches or existing target branches, previews candidate repositories in a table, then prompts once unless `--yes` is set. Declining the confirmation is a successful skip/no-op. Applying the command renames only local branches.

### `untrack`

`untrack` scans repositories with `.gitignore` files for tracked ignored files. It previews candidate repositories, prompts once unless `--yes` is set, then runs `git rm --cached` for ignored tracked files. It leaves index changes staged for the user and does not create commits.

### `reset`

`reset` fetches `origin <current-branch>` with the remote timeout and transient retry policy, skips detached HEAD and missing remote counterparts, previews candidates in a table with ahead/behind and dirty state, and then runs `git reset --hard origin/<branch>` after one aggregate confirmation.

### `remove-secrets`

`remove-secrets` refreshes `origin` before scanning history for a fixed set of sensitive filename/path patterns. Any fetch failure stops before scanning or mutation. With `--no-fetch`, it warns that local remote-tracking refs may miss remote-only commits before scanning. It prints matched files for affected repositories, counts clean repositories, and only rewrites repositories with matches. It always passes `--partial --force` to `git-filter-repo`.

Before mutating a repository, `remove-secrets` deletes `.git/git-wrangler/baseline/` and clears managed rewrite metadata that cannot safely cross secret removal. It does not create a replacement baseline. The next `rewrite-authors`, `rewrite-commits`, `rewrite-dates`, or `rewrite-hours` run starts a fresh rollback baseline from the already-cleaned history.

### `rewrite-authors`

`rewrite-authors` refreshes `origin` before previewing the author rewrite. Any fetch failure stops before the destructive notice, confirmation, or mutation. With `--no-fetch`, it warns that local remote-tracking refs may miss remote-only commits before continuing. `--rewrite-after` and `--rewrite-before` select local branch commits by their current author date. Repositories with no selected commits are successful skips.

Before mutation, it adds missing entries for selected commits to the shared rewrite baseline under `.git/git-wrangler/baseline/`. Existing baseline entries keep their first-baselined metadata. After a successful `git-filter-repo` run, only current SHA mappings are updated from `.git/filter-repo/commit-map`.

### `rewrite-commits`

`rewrite-commits` validates the configured AI endpoint, model, and credentials before checking dependencies or refreshing `origin`, then scans repositories. Any fetch failure stops before scanning, AI requests, preview, confirmation, or mutation. With `--no-fetch`, it warns that local remote-tracking refs may miss remote-only commits before scanning and before the normal AI data-send confirmation path. `--rewrite-after` and `--rewrite-before` first select local branch commits by current author date; out-of-range commits are not sent to the AI endpoint and are not rewritten. `--skip-conventional` excludes commits whose first line already parses as a Conventional Commit; `--require-scope` implies it and tightens the rule so only commits with both a type and a scope are skipped. With `--require-scope`, generated subjects must also include a scope: the prompt demands one and scopeless output is rejected and retried like other invalid messages, so a rewritten commit always passes the same filter that selected it. Auto-generated commits are never sent to the AI endpoint and are never rewritten: merge commits (two or more parents) plus Git-generated `Revert "..."`/`Reapply "..."` subjects and `fixup!`/`squash!`/`amend!` autosquash markers. They are reported as "Skipped auto-generated commits" in the data-send notice and summary. Generation may prompt before sending data; applying generated messages is a separate all-repository confirmation. Old commit messages are not sent as model context. The AI context includes file paths, stats, a compact change summary, and redacted diff snippets; repository metadata, summaries, file lists, and stats are preserved ahead of diff text when internal context bounds truncate input. Generation requests use JSON object response format and validate every returned message. `--batch-size` is a maximum commits-per-request value; dynamic packing may send smaller batches to stay within bounded request context. Requests run through a shared pooled HTTP/1.1 client with at most `--concurrency` requests in flight while `--rpm` paces request starts. Provider context-limit, malformed JSON, truncated-output, transient transport, HTTP 429, and HTTP 5xx failures are retried up to four batch attempts with jittered exponential backoff; remaining failures get a final single-commit recovery pass with smaller context before the command aborts. Individual retry attempts are not printed; transient retries are aggregated into one end-of-run summary line, while per-commit escalation and final-recovery notices still print immediately. For better high-level messages on broad changes, use `--body`, reduce `--batch-size`, or choose a stronger configured model. Declining either confirmation is a successful no-op before mutation. Final apply output is aggregate: rewritten commit messages, updated repositories, and failures.

Before applying generated messages, it adds missing shared-baseline entries only for commits whose messages will actually change. Descendant commits that change SHA only because their parent changed are replayed by `rollback-rewrites` when needed.

### `rewrite-dates`

`rewrite-dates` refreshes `origin` before preparing timestamp rewrites. Any fetch failure stops before scanning, planning, preview, prompt, or mutation. With `--no-fetch`, it warns that local remote-tracking refs may miss remote-only commits before scanning.

Without `--repo`, all discovered repositories are planned together. With `--repo`, exact single-repository targeting is used. Only commits reachable from local branch refs are considered and only local branch refs are passed to `git-filter-repo` through `--refs`; internal `refs/git-wrangler/*` refs are excluded from rewrite refs. Repositories with fewer than two local-branch commits are skipped for normal rewrites. Repositories with no commits selected by `--rewrite-before` / `--rewrite-after` are skipped. Normal rewrites require a clean working tree and fail that repository before mutation when it is dirty.

`--rewrite-after` and `--rewrite-before` select commits by current author date, not by first-baselined/original author date. `--start-date` and `--end-date` define the target range, not the commit-selection range. If one target bound is omitted, it is inferred globally from selected current author dates. If both are omitted, the global selected current min/max range is used. `--days N` instead targets the last N days ending on `--until YYYY-MM-DD`, or today when `--until` is omitted, and cannot be combined with `--start-date` or `--end-date`.

`--window HH:MM-HH:MM` keeps the existing calendar, active/rest day, quota, seed, frequency, spread, and topology planning, but replaces generated per-day working hours with the supplied same-day window for every day. `--window` also accepts comma-separated day-of-week schedule entries such as `mon-fri=16:00-20:00,sat=10:00-14:00`; days omitted from the schedule keep the generated per-day working-hour behavior. Date redistribution and explicit windowing happen in one `git-filter-repo` pass.

Planning is global across all candidate repositories. With `--repo`, the same planner runs with one candidate repository. The planner first builds one run-level activity calendar in the planning timezone, classifying each target day as inactive, rest, active, or forced-active, then fits selected commits into exact daily quotas from that calendar in global current-author-date order. One planning timezone is used per run: the dominant current author timezone among selected commits, or the local offset when no selected commit has a valid timezone. That timezone is used for target-date parsing, preview dates, and rewritten author/committer callback dates. Generated timestamps stay on the same UTC activity date as the planning date when possible, so shared rest dates remain blank in `activity` output unless they are forced-active.

Parent/child ordering is enforced per repository with at least one second between selected relatives. Unselected commits act as fixed date boundaries. Commit volume is absorbed by larger quotas on already active days and never activates rest days by itself. Topology or fixed boundaries may activate inactive days or convert the minimum required rest days to forced-active. Explicitly impossible target ranges fail before mutation with the repository and commit named; inferred ranges may expand minimally to satisfy ordering.

`--frequency` controls the generated calendar shape. Active-day density scales dynamically from selected commits, available non-rest days, and frequency; there is no fixed daily capacity or maximum. `low` creates sparse work sessions with long gaps, lower weekend active-day likelihood, shorter workdays, and longer vacation- or holiday-like pauses. `medium` is the clustered weekday-heavy default with multi-day gaps and moderate rest periods. `high` creates denser sessions with shorter gaps, stronger multi-day bursts, higher weekend active-day likelihood, and more evening activity. When a plan has multiple sessions, they are distributed broadly across the complete target range without guaranteeing activity on the exact start or end dates. Frequency necessarily changes average commits per active day because the selected commit count is fixed.

`--spread` controls how evenly selected commits are allocated across active days after the calendar is selected. `low` keeps daily quotas more even, `medium` is the default moderate variance, and `high` allows larger day-level and session-level quota swings with stronger consecutive-day correlation. Spread does not directly select active days, and weekend active-day likelihood is not reused as a workload quota weight. Final active or forced-active days may still differ across spread settings after fixed-date and topology enforcement adjust a complete plan.

For longer target ranges, the planner adds synthetic vacation- or holiday-like pauses before selecting work sessions. These pauses are generated from the seed, are deterministic, and are only applied when the target range and selected commit count are large enough. Consecutive active days form sessions with seeded session-level workload multipliers and correlated day-level variation. Effective repository concentration modestly damps aggregate variation when selected commits are balanced across many repositories without equalizing repository counts. Every active day receives one commit, then remaining commits are distributed proportionally by workload weight with seeded largest-remainder allocation. Daily quotas always sum exactly to the selected commit count.

Retry attempts `0` through `7` rebuild the whole calendar with the existing salted seed rule. Topology-correct attempts are ranked by forced-active days, remaining topology compression, maximum one-second run, one-second edges, daily quota deviation after topology enforcement, and adjusted commits. The existing compression warning is shown when the chosen result remains compressed.

The planner seed comes from `--seed`, then a generated seed. Preview output shows the seed source, planning timezone, active days, rest days, forced-active days, selected commit count, filters, frequency, spread, time window, and final post-topology median, p90, and maximum commits per active day.

Before mutation, `rewrite-dates` adds missing entries for selected commits to the shared rewrite baseline under `.git/git-wrangler/baseline/`. Baseline entries are command-neutral and store the first-baselined SHA, latest current SHA, tree, first-baseline parents, author and committer identity/dates, full message, and the bundle layer containing the original object. Existing baseline metadata is not overwritten. After a successful rewrite, `.git/filter-repo/commit-map` updates only current SHA mappings and SHA aliases needed to reconnect later baseline generations.

`rewrite-dates --rollback` was removed. Use `rollback-rewrites` to restore the shared baseline created by `rewrite-authors`, `rewrite-commits`, `rewrite-dates`, and `rewrite-hours`.

### `rewrite-hours`

`rewrite-hours` refreshes `origin` before preparing timestamp rewrites. Any fetch failure stops before scanning, planning, preview, prompt, or mutation. With `--no-fetch`, it warns that local remote-tracking refs may miss remote-only commits before scanning.

`--window` is required and must be either one same-day `HH:MM-HH:MM` window with the start before the end, or a non-empty comma-separated day-of-week schedule such as `mon-fri=16:00-20:00,sat=10:00-14:00`. The command considers commits reachable from the same local branch ref surface used by `rewrite-dates`; `--rewrite-after` and `--rewrite-before` first select commits by current author date. It then preserves each selected commit's current calendar date in the run planning timezone and rewrites both author and committer dates into the scheduled window for that day. Timestamps are deterministic, sorted, and second-resolution; duplicate seconds are allowed only when a day's commit count exceeds the number of seconds in the window.

Plain `--window HH:MM-HH:MM` is applied uniformly to every targeted repository and every day. Schedule entries apply only to their listed days; commits on omitted days keep their existing timestamps and are excluded from the date callback and baseline capture. Parent/child timestamp ordering is validated before mutation. The command creates or extends the same shared baseline as `rewrite-authors`, `rewrite-commits`, and `rewrite-dates`, then applies one `git-filter-repo` date callback per repository and updates current SHA mappings from `commit-map`.

### `rollback-rewrites`

`rollback-rewrites` requires `git`, does not require `git-filter-repo`, does not fetch, and skips repositories with no shared baseline. It reads `.git/git-wrangler/baseline/manifest.json` and bundle layers from `.git/git-wrangler/baseline/bundles/`. Invalid or incomplete baseline storage fails the repository before mutation.

Rollback requires a clean working tree before restoring any repository. It previews local branch refs that contain baselined commits, reports baselined and unbaselined commits on affected branches, warns that rollback is local-only, and prompts once. Remote propagation requires `git-wrangler push --force`.

For each affected local branch, rollback rebuilds commits in topological order. Baselined commits restore the stored tree, message, author, committer, and dates, with parent SHAs remapped to restored or replayed parent results so later baseline generations connect to earlier restored commits. Unbaselined commits are left as existing objects when parent links do not change; they are replayed with their current tree, message, author, committer, and dates only when parent links must be reconnected. Rollback may produce new SHAs for replayed commits and may not preserve signatures when replay is required.

On successful repository rollback, affected branch refs move to rebuilt heads, the checked-out worktree/index resets to restored `HEAD`, shared baseline storage is cleared, and managed rewrite metadata that would be stale after rollback is cleared. Tags keep the normal rewrite-command warning behavior; local branches are the rollback surface. Commits created after an earlier baseline are preserved and replayed only if needed to reconnect them to restored parents.

### `init`

`init` owns GitHub device authentication. It prints the one-time code and verification URL before an explicit Enter prompt through the shared prompt session, attempts to open the URL with the platform browser launcher, and continues waiting when no browser is available. The server-provided device-code expiration is the authentication timeout, and command cancellation stops the prompt or wait immediately.

GitHub credentials resolve only from `GIT_WRANGLER_GITHUB_TOKEN` or Git Wrangler's keyring account. `GH_TOKEN` is used only as an outbound transport variable when Git Wrangler invokes `gh`; inbound `GH_TOKEN` is ignored.

`init` checks keyring availability before secret-related prompts. When unavailable, it skips GitHub OAuth and AI API-key entry, still saves prompted GitHub host and AI provider/base URL/model values, and exits successfully. Warnings explain that secure credential storage is unavailable and secret setup was skipped without exposing backend keyring errors. Missing GitHub auth is reported with `GIT_WRANGLER_GITHUB_TOKEN` guidance. Missing AI auth is reported with `GIT_WRANGLER_AI_API_KEY` guidance, plus `OPENAI_API_KEY` for the OpenAI provider.

### `config`

`config show` prints non-secret key/value sections. `config set` accepts plaintext values for non-secret keys only. Secret keys (`github.auth`, `ai.api-key`) must be entered through the prompt and are stored through the credential store. AI gateway headers use `ai.headers.<name>`: inline values are stored in config, while omitted values prompt for a secret and store it in the credential store. `config unset` removes stored secrets and AI header values.
